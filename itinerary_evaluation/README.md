# 📍 LLM-Travel-Grounding-Evaluator

> **A Rule-Based Database Grounding Framework for Validating the Spatiotemporal Consistency of LLM-Generated Travel Itineraries**

![Python](https://img.shields.io/badge/Python-3.8%2B-blue?logo=python)
![SQLite](https://img.shields.io/badge/SQLite-3-003B57?logo=sqlite)

An automated evaluation framework designed to detect **hallucinations, scheduling conflicts, and spatial inefficiencies (backtracking)** in travel itineraries generated by Large Language Models (LLMs). By grounding LLM outputs against a verified relational database (SQLite), this project provides objective quality metrics for generative AI planners.

---

## 💡 Key Features & Validation Metrics

The core engine automatically parses and cross-references LLM responses across three distinct validation layers:

1. **Foundational Integrity (`check_quality`)**
   * **Hallucination Check:** Identifies whether the LLM generated fictional landmarks or places absent from the reference database.
   * **Duration Enforcement:** Calculates total visiting time (`spent_time`) and triggers an alert if it exceeds a realistic daily limit of **8 hours**.

2. **Temporal Consistency (`closing_day_check`)**
   * Converts the planned itinerary date into a weekday index and queries the database to ensure the LLM hasn't scheduled a visit on a location's **closing day** (e.g., trying to visit the Louvre on a Tuesday).

3. **Geospatial Efficiency (`check_route_efficiency`)**
   * Sequences destinations chronologically and evaluates sequential pairs ($Spot_{i} \rightarrow Spot_{i+1}$) using real-world transit distance (`distance_km`) and duration (`transit_min`).
   * Flags highly inefficient, "zig-zag" routing (backtracking).
   * Features a robust **guard clause** (`len(itinerary) <= 1`) to gracefully handle single-spot days without unbinding errors.
